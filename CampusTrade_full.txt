2. System Architecture
2.1 High-Level Architecture
CampusTrade follows a three-tier client-server architecture. The React SPA communicates exclusively through a versioned REST API (/api/v1/*) served by Express. Socket.io is mounted on the same Node process and uses a separate namespace (/barter) for real-time negotiation events.

Tier | Component | Responsibility
Presentation | React SPA | Renders UI, manages local state, connects to REST + WS
Application | Express API Server | Business logic, auth middleware, REST routing
Real-time Layer | Socket.io (same process) | Barter negotiation rooms, presence events
Persistence | PostgreSQL | Relational data: users, items, bookings, reviews
Object Storage | S3 / Cloudinary | Item images; signed URLs served via API

2.2 Request Flow
Client sends HTTPS request to Express router.
Auth middleware validates JWT from Authorization: Bearer header.
Route handler calls service layer which executes parameterised SQL / ORM query.
Response serialised to JSON; images referenced as CDN / signed S3 URLs.
For barter rooms: client connects via Socket.io using the same JWT; server validates on the connect event.

2.3 Environment Configuration
All secrets must be stored in environment variables, never committed to the repository.
DATABASE_URL — PostgreSQL connection string
JWT_SECRET — HS256 secret (minimum 256-bit entropy)
JWT_REFRESH_SECRET — separate secret for refresh tokens
AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / S3_BUCKET — or CLOUDINARY_URL
SMTP_HOST / SMTP_USER / SMTP_PASS — for verification emails
CLIENT_ORIGIN — CORS whitelist (e.g., https://campustrade.app)

4. Authentication & Authorization
4.1 Campus Restriction Implementation
On registration, extract the domain from the submitted email.
Reject if domain is not in ALLOWED_DOMAINS array (configurable; defaults to *.edu).
On acceptance, generate a 6-digit OTP, hash it (SHA-256), store with 15-minute expiry, and send via Nodemailer.
The /auth/verify endpoint accepts the OTP, validates hash + expiry, sets email_verified = TRUE.
All protected routes require email_verified = TRUE (checked in middleware).

4.2 JWT Strategy
Token | Expiry | Storage | Rotation
Access Token | 15 minutes | Memory (React state / context) | Issued alongside each refresh
Refresh Token | 7 days | httpOnly Secure SameSite=Strict cookie | Rotated on every /auth/refresh call

Refresh token stored in the database (hashed) for revocation support.
On logout, delete refresh token from DB and clear cookie.

4.3 Password Requirements
Minimum 8 characters, at least one uppercase, one digit, one special character.
Hashed with bcrypt cost factor 12 before storage; never logged or transmitted in plaintext.

6. Core Business Logic
6.1 Temporal Booking — Overlap Prevention
The availability check must be atomic to prevent race conditions. Implementation strategy:
Use PostgreSQL's daterange overlap exclusion constraint (Section 3.2.3) as the last-resort guard.
Before insert, run a SELECT query with FOR UPDATE SKIP LOCKED to acquire a row-level advisory lock on the item_id.
If any booking exists with status IN ('Pending','Active') and daterange overlaps the requested range, return HTTP 409 Conflict with a descriptive error.
Frontend calendar greys out dates by pre-fetching /items/:id/availability on component mount and on month navigation.

6.2 Barter Flow State Machine
State | Trigger | Next State | Side-Effect
— | Borrower clicks 'Propose Barter' | Pending (Barter) | Booking row inserted; WS room opened
Pending | Owner clicks 'Accept Trade' | Active (Barter) | booking.payment_type = Barter; system message sent
Pending | Either party disconnects / 24h timeout | Cancelled | WS room closed; booking deleted
Active | Owner marks item returned | Completed | Trust score recalculated for both users

6.3 Trust Score Algorithm
Recalculated server-side after every booking completion or penalty event. The formula from the PRD expressed as SQL / pseudocode:
T_score = (W1 * R_completed/R_total) + (W2 * L_completed/L_total) - P_penalties

Variable | Meaning | Value / Source
W1 | Weight for review ratio | 40
W2 | Weight for successful rental ratio | 60
R_completed / R_total | Proportion of reviews with rating >= 4 | Queried from reviews table
L_completed / L_total | Proportion of bookings with status = Completed | Queried from bookings table
P_penalties | Fixed deduction per penalty event | 5 points per flagged event

Score is clamped between 0 and 100.
New users start at 50. Users with fewer than 3 transactions display 'New User' badge.
The recalculation is triggered asynchronously (background job / Promise.all) to avoid blocking the HTTP response.


7. Real-Time WebSocket Specification
7.1 Socket.io Configuration
Namespace: /barter
Transport: WebSocket with polling fallback
Auth: JWT passed in socket.handshake.auth.token; validated in middleware before connection is permitted
CORS: restricted to CLIENT_ORIGIN

7.2 Events Reference
Client → Server
Event | Payload | Description
join_room | { booking_id } | Join barter room; server validates user is lender or borrower
send_message | { booking_id, content } | Broadcast message to both parties
accept_trade | { booking_id } | Lender only; triggers booking status update + system message
reject_trade | { booking_id } | Either party; cancels booking + closes room

Server → Client
Event | Payload | Description
room_joined | { booking_id, history: [] } | Sends last 50 messages on join
new_message | { message_id, sender_id, content, sent_at } | Real-time message broadcast
trade_accepted | { booking_id, barter_terms } | Both clients receive; triggers UI state change
trade_rejected | { booking_id } | Room cleanup signal for both clients
user_typing | { user_id } | Throttled typing indicator

7.3 Message Persistence
Every send_message event is persisted to barter_messages before broadcasting.
On room_joined, server fetches the last 50 messages from the DB and sends as history.


8. File Upload Specification
8.1 Flow
Frontend sends POST /items with Content-Type: multipart/form-data.
Express uses Multer (memory storage) to buffer files; validates MIME type (image/jpeg, image/png, image/webp) and max size (5 MB per file, max 5 files).
Backend uploads to S3 / Cloudinary using server-side SDK with a generated UUID key.
Resulting CDN URLs stored in items.image_urls array.

8.2 Security Controls
Signed URLs with 1-hour expiry for private items (S3 presigned URL pattern).
File type validated by magic bytes (not just extension) using file-type npm package.
Original filenames never stored or used in keys to prevent path traversal.


9. Security Requirements
Threat | Control | Implementation Detail
SQL Injection | Parameterised queries | Sequelize / Knex — never string-interpolate user input
XSS | Output encoding + CSP header | React escapes by default; set Content-Security-Policy header
CSRF | SameSite cookie + CSRF token | Refresh token cookie: SameSite=Strict; state-changing WS events require valid Bearer token
Brute Force | Rate limiting | express-rate-limit: 10 req/min on /auth/login and /auth/verify
Enumeration | UUID PKs | No sequential IDs exposed in URLs
Unauthorised Access | Ownership middleware | Middleware checks req.user.id === resource.owner_id before mutation
Insecure Upload | MIME + size validation | Magic-byte check + Multer limits (Section 8.2)
Exposed Secrets | Environment variables | dotenv for local; secrets manager in production

9.1 HTTP Security Headers
Helmet.js applied globally: sets X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security, and Referrer-Policy.
CORS: explicitly whitelist CLIENT_ORIGIN; reject all other origins.

10.2 State Management
Auth state: React Context + useReducer; access token stored in memory (never localStorage).
Item listings: local useState with SWR or React Query for caching + revalidation.
Barter room: Socket.io client events update local component state; no global store needed.

10.3 Calendar Component
Use react-calendar or build with date-fns; no heavyweight library required.
On mount, fetch /items/:id/availability and disable all dates within returned ranges.
Validate selected range client-side before submitting booking to prevent unnecessary round-trips.
Dynamically calculate total price: (end_date - start_date + 1) * daily_price, displayed in real time.
